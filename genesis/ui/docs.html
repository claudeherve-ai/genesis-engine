Build a multi-agent system anywhere.

Genesis Engine turns one sentence into a tested, deployed multi-agent system. This guide gets you running locally, in Docker, on Azure, or on Kubernetes — and runs green offline with zero API keys.

🧬 What is Genesis Engine?

You describe a problem in plain English. Genesis runs a five-stage pipeline and hands you a working system:

🔍

Analyze

Decomposes your problem into a domain model with grounded citations.

🏗️

Architect

Designs agent topology + concrete test scenarios with expected outputs.

🔧

Build

Writes prompts and verifies every tool against the live tool catalog — no hallucinated tools.

🧪

Test

Actually executes agents in a sandbox against the scenarios and scores from reality.

🚀

Deploy

Provisions on your target (AgentSystem, Docker, Azure, K8s, local) with health checks.

📊

Observe

Token/cost telemetry, transcripts, citations, versioning, and rollback.

Anti-hallucination by design. Agents may only use tools from the verified catalog, grounded in Microsoft Learn + live web search. Every claim carries a citation.

⚡ 60-second quickstart

The fastest path. Works with no API keys (offline dry-run deploy). Add keys later for real LLM + deploy.

Then open http://localhost:8000, type a sentence, and click Build my system.

Prefer not to install? Jump to Docker for a one-command container.

🛠️ Build → test → deploy in the UI

1
Start from a recipe — scroll to the Recipe gallery and click a template (Support, Research, DevOps, etc.), or write your own description.
2
Pick a target & (optionally) require approval — choose Docker / Azure / K8s / Local, and tick Require human approval for a pause-before-deploy gate.
3
Watch it build live — the pipeline streams each stage. When done you'll see the test score, real execution transcripts, citations, and cost.
4
Deploy or export — click Deploy this system for copy-paste steps per environment, or Export agents to download JSON + Markdown.

🔐 Auth, RBAC & secrets

Lock down the endpoint

Set GENESIS_API_KEYS to require an X-API-Key header. Roles are admin (full access incl. approve/rollback) and user (build/read).

When GENESIS_API_KEYS is unset, the API is open — perfect for local dev and the test suite, but set keys before exposing it.

🛜 Offline mode

With no AZURE_OPENAI_API_KEY, Genesis runs in a safe offline mode: the deploy stage performs a dry-run (health-checks pass, no external calls), web search falls back to DuckDuckGo, embeddings use a deterministic TF-IDF fallback, and the full test suite passes. This makes it perfect for demos, CI, and air-gapped evaluation.

Offline builds won't produce real LLM-authored prompts. Add AZURE_OPENAI_API_KEY for production-quality generation and a real deploy target for live endpoints.

🩺 Troubleshooting

The build fails immediately

Offline you'll get dry-run deploys, not failures. If a build fails, check the server logs and confirm AZURE_OPENAI_API_KEY is valid when generation is expected.

403 / 401 responses

Auth is enabled via GENESIS_API_KEYS. Pass X-API-Key; admin-only routes (approve, rollback) need the admin key.

429 Too Many Requests

The rate limiter kicked in. Raise or disable it with GENESIS_RATE_LIMIT (set 0 to turn off).

Templates / metrics panel empty

Confirm the server is running and reachable; the UI fetches /v1/templates and /v1/metrics from the same origin.
